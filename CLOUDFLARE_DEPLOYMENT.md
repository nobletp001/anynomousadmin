# Cloudflare Workers deployment report

## Branch model

- `main` remains the Netlify branch.
- `cloudflare-admin` is the only branch configured to deploy this admin app to Cloudflare Workers.
- Do not merge `cloudflare-admin` back into `main` unless the Netlify/Cloudflare split is intentionally redesigned.

## Project inspection

- Next.js version: `16.2.6`.
- Router type: App Router under `src/app`; no `pages` directory.
- Rendering model: client-heavy admin dashboard with Server Component layouts and client page components. No API route handlers were found.
- Server Components: `src/app/layout.tsx` is a Server Component; most dashboard/login pages declare `"use client"`.
- Server Actions: none found.
- Middleware: none found.
- API routes: none found in this repo. API calls go to the Railway backend through `NEXT_PUBLIC_API_URL`.
- Authentication storage: the frontend stores admin access/refresh tokens in `sessionStorage` and sends credentials with Axios `withCredentials: true`.
- Authentication cookies: no cookies are created by this frontend. Railway should set backend cookies as `HttpOnly`, `Secure` in production, and explicit `SameSite` according to cross-site needs.
- Image optimization: no `next/image` usage found. Images are mostly remote URLs rendered directly, including Cloudinary-hosted proof images.
- Node.js-only dependency review: `xlsx` is used in browser-triggered Excel exports; report/PDF generation uses browser windows. No repo usage of `fs`, `net`, `tls`, `child_process`, `worker_threads`, native binaries, local filesystem writes, or long-running server processes was found.

## Cloudflare Worker

- Worker name: `payfluence-admin`.
- Worker entry point: `.open-next/worker.js`, generated by OpenNext.
- Adapter: `@opennextjs/cloudflare`.
- Wrangler config: `wrangler.jsonc`.
- OpenNext config: `open-next.config.ts`.
- Runtime flags: `nodejs_compat`, `global_fetch_strictly_public`.
- Static assets binding: `ASSETS` mapped to `.open-next/assets`.
- Observability: enabled in `wrangler.jsonc`.
- Current deploy target: `workers.dev`.
- Custom domain to connect later: `admin.payfluence.com`. Add the route/custom domain in Cloudflare after the `payfluence.com` zone is active in Cloudflare; this repo does not change DNS automatically.
- Bindings: only the required static assets binding and `WORKER_SELF_REFERENCE` service binding are configured. No R2, D1, KV, or Durable Object cache bindings were added because this app does not currently use ISR or on-demand revalidation.

## Environment variables and secrets

Required GitHub Actions secrets:

- `CLOUDFLARE_API_TOKEN`: scoped Cloudflare API token for Workers deploys. Do not use the Global API Key.
- `CLOUDFLARE_ACCOUNT_ID`: Cloudflare account id.
- `NEXT_PUBLIC_API_URL`: Railway backend API URL, including `/api`. Current value for this deployment is `https://anynomousbackend-production.up.railway.app/api`.

Local setup:

```bash
cp .env.example .env.local
# set NEXT_PUBLIC_API_URL to the real Railway API URL including /api
```

The production build fails clearly when `NEXT_PUBLIC_API_URL` is missing. It is set in `wrangler.jsonc` for Wrangler/OpenNext builds, and any other production build environment must also set it. Do not put private secrets behind a `NEXT_PUBLIC_` prefix.

## Railway backend requirements

- Keep the backend on Railway.
- Add the final Cloudflare admin origin, `https://admin.payfluence.com`, to the Railway backend CORS allowlist.
- Credentialed CORS must set a specific allowed origin and `Access-Control-Allow-Credentials: true`; do not use `*` with credentials.
- Login, logout, refresh, password reset, payment, wallet, report, and private admin API responses should use `Cache-Control: no-store`.
- Backend-set cookies should be `HttpOnly` where JavaScript does not need to read them, `Secure` in production, and `SameSite=None` only if the final frontend/backend domains are cross-site and credentials must be sent cross-site.
- Do not log passwords, tokens, cookies, authorization headers, API secrets, payment data, wallet data, private reports, or personal data.

## Netlify requirements

This branch adds a `netlify.toml` ignore command that skips Netlify builds when `BRANCH=cloudflare-admin`.

Manual Netlify dashboard setting still required:

- Keep production branch set to `main`.
- Disable branch deploys for `cloudflare-admin`.
- Disable deploy previews for pull requests from `cloudflare-admin`, or restrict deploy previews to approved branches that do not include `cloudflare-admin`.

Expected behavior:

- Push to `main`: Netlify deployment only.
- Push to `cloudflare-admin`: Cloudflare Workers deployment only.

- `Content-Security-Policy`

The CSP is built from current usage:

- Google Fonts for the report print view.

## Cloudflare caching

- Static Next assets under `/_next/static/*` are cached as immutable for one year through `public/_headers`.
- Versioned JavaScript, CSS, fonts, and known public immutable SVG assets are cached as immutable.
- Private admin pages, dashboard HTML, login/session content, payment data, wallet data, reports, and API responses are not intentionally cached.
- No global Cache Everything rule should be created.

## Cloudflare security recommendations

- Enforce HTTPS.
- Use Full Strict SSL.
- Put Cloudflare Access in front of the admin domain for approved administrators, but keep the app's own authentication and authorization.
- Enable Cloudflare WAF managed rules.
- Add rate limiting for login and password-reset routes.
- Enable bot protection for the admin domain.
- Restrict Cloudflare API tokens to the minimum Workers deployment permissions required.

## Rollback

1. In Cloudflare Workers, roll back `payfluence-admin` to the previous Worker version.
2. If needed, revert the latest commit on `cloudflare-admin` and push the branch.
3. Keep `main` unchanged so the Netlify deployment path remains available.
